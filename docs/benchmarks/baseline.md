# Benchmark Baseline — ratatui-ppalla prepared primitives

> First quantitative baseline for the PreparedText / PreparedLayout primitives
> (ROADMAP Direction 2). Measured with `criterion` on the prepared benchmark
> suite: `cargo bench --bench prepared`.

## Environment

- **Date:** 2026-07-22
- **Toolchain:** stable (MSRV 1.74 declared)
- **Measurement:** `criterion 0.5` (3s warmup, 100 samples)
- **Input:** deterministic 1000-line × 80-column ASCII text generated by
  `make_lines(1000, 80)` in `benches/prepared.rs`. ASCII-only isolates the
  layout/wrap hot path (grapheme segmentation is trivial for ASCII).

## Analysis

### ✅ Win: PreparedLayout cache hit (64.7 ns vs 10 000 ns target)

The 1-entry constraint-evaluation cache exceeds its target by **155×**. On a
cache hit the hot path is a single `DefaultHasher` signature comparison + mutex
lock (uncontended single-threaded), with no ratatui `Layout::split` call. This
validates the prepare/layout separation for layout regions.

### ❌ → ✅ Resolved: PreparedText::layout (windowed cloning)

**Resolved.** The flagship text layout was 5.7× over the acceptance target and
23× slower than ratatui's `Paragraph` + `Wrap`. The fix deferred grapheme
`String` cloning to the visible window only: the hot path still walks ALL
logical lines to compute `total_lines` (needed for scroll clamping), but instead
of cloning each segment into an owned `Vec<TextSegment>` for every display line,
it accumulates cheap `&TextSegment` borrows and materializes owned segments only
when a display line's global index lands inside `[scroll, scroll+height)`.

Result: **2.83 ms → 0.194 ms (14.6× faster)**, meeting the ≤ 0.5 ms target with
2.6× headroom. Correctness is guaranteed by a differential test
(`differential_layout_matches_reference_over_matrix`) that checks 1 080 input ×
width × scroll × height combinations against a brute-force reference, including
mid-logical-line window splits. The `Preparable` trait was NOT changed.

The original root-cause analysis is preserved below for reference.

### Original root-cause analysis (pre-fix)

The hot path allocated per frame. For every visible and non-visible display line
it pushed `seg.clone()` (a `String` clone per grapheme) into `Vec<TextSegment>`s.
For 1000 lines × ~80 graphemes that was ~80 000 `String` clones + `Vec` growth
per frame. The `Preparable` trait's `type Layout` is a non-GAT associated type,
so the layout result must *own* its data rather than borrow the prepared state —
which forced the cloning for the *returned* visible lines. (The non-visible
clones were pure waste and are what the fix eliminated.)

This was precisely the kind of regression the benchmark infrastructure exists to
catch: "can't optimize what you can't measure."

## Optimization directions

1. **GAT layout result** — make `type Layout<'a>` borrow the prepared state so
   `layout` returns `&'a`-references to graphemes instead of cloning. Requires
   lifting the trait to use generic associated types (stable since Rust 1.65,
   within MSRV 1.74). Would eliminate the remaining visible-window clones too
   (currently ~1 900/frame); further headroom beyond the now-met target.
2. ~~**Windowed walk**~~ — **✅ DONE (this pass).** `layout` now clones segments
   only for the visible `[scroll, scroll+height)` window; all other lines are
   walked width-only to compute `total_lines`. Brought `layout_text/1000x80`
   from 2.83 ms to 0.194 ms (14.6×).
3. **Avoid re-wrapping unchanged width** — cache the wrapped display-line
   structure in the prepared state keyed by width (like `PreparedLayout`'s
   1-entry cache), so repeated frames at the same width are pure windowing.
   Would make repeated same-width frames even cheaper than the current 194 µs.
4. **Small-string optimization for graphemes** — replace `String` in
   `TextSegment` with a small inline buffer (e.g. `compact_str` / hand-rolled
   `[u8; 4]`) to remove heap allocation entirely. Most graphemes are ≤ 4 bytes.
   Would also speed up the cold `prepare` path.

Items 3 and 4 are now optional polish — the acceptance target is already met.

## Reproducing

```bash
cd ~/source/project/ratatui-ppalla
cargo bench --bench prepared
```

Results vary by hardware; the *ratios* (cache hit vs miss, layout vs Paragraph)
are more stable than absolute times across machines.
